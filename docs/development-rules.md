# 项目开发守则

## 使用方式

- AI 在本项目开始实质性工作前，必须先阅读本文件。
- 如果本文件与临时对话要求冲突，以用户在当前对话中的明确要求为准。
- 本文件由用户规定，AI禁止修改本文件

## 开发原则

- 代码开发中，应包含必要注释与日志（注释用中文，日志用英文）
- 代码必须要写测试
- 优先保持模块边界清晰，避免跨模块直接耦合业务细节
- 优先通过接口和策略模式保留未来替换实现的空间

## 代码约定

- Prompt、结构化输出 schema、模型调用逻辑必须集中管理，不得散落在业务模块中
- 领域对象与流程控制分离：状态归领域层，执行流程归 runtime / workflow 层
- 导演权限逻辑不得散落到普通社交模块，应通过统一权限策略层处理

## 架构与变更约束

- V1 不引入 LangChain/LangGraph 作为核心依赖
- 必须保留以下扩展点：
  - ThinkingEngine：用于切换 AI 思维方式
  - PermissionPolicy：用于切换用户权限模型
  - WorkflowRunner：用于未来接入 LangGraph 或其他复杂编排器
  - LLMClient：用于隔离具体模型供应商 SDK
- 新增功能时优先落在已有领域模块内，只有当职责边界明确时才新增模块
- 不向最终用户暴露模型原始 chain-of-thought

## 交付要求

- 完成开发后，更新 `docs/project-progress.md`与 `docs/project-context.md`。
